# ส่วนที่ 1 — โมดูล E-SIGNATURE (ลงนามดิจิทัล / ลงนามด้วยมือ)

## 1.1 โครงสร้างข้อมูล
```js
Document = {
  id, title,
  creatorId,
  date, ts,                 // ts = วันที่ในเดือน ใช้กรองตามช่วงเวลา
  docType,                  // 1 ใน 11 ประเภทหลัก (legacy): "เอกสารทั่วไป", "เอกสารลับ", "เอกสารบุคลากร", "สัญญาธุรกิจ/MOU" ฯลฯ
  docSubtype,                // ประเภทย่อย (2-layer catalog ตั้งค่าได้ที่ Tab 6)
  docPrefix,                 // prefix สำหรับสร้างเลขที่เอกสาร
  docNo,                     // เลขที่เอกสาร รูปแบบ "PREFIX-ปี/ลำดับ" เช่น FIN-2026/001
  otherTypeName?,             // ถ้าเลือก "อื่นๆ" จะพิมพ์ชื่อเอง
  files: [{ id, name, pages, file }],   // ไฟล์ PDF หลัก
  attachments: [{ name, file }],          // ไฟล์แนบเสริม ไม่ต้องเซ็น
  cc: [userId],                            // ผู้ที่ถูก CC ดูได้อย่างเดียว
  signers: [{
     id, step, role: 'signer'|'approver', status: 'pending'|'viewed'|'signed'|'rejected',
     assignedTo?, time?, reason?, sigType?
  }],
  placements: [{ id, fileId, page, xPct, yPct, signerId, sig? }], // ตำแหน่งช่องเซ็นบน PDF
  signedVersions?: [{ files, byId, time }],  // ฉบับที่เซ็นมือ/สแกน แยกเก็บ ไม่ทับไฟล์ต้นฉบับ
  comments: [...],
  status: 'progress' | 'done' | 'rejected' | 'cancelled',
}
```

## 1.2 ประเภทของผู้เกี่ยวข้อง (role)
| role | มีช่องเซ็นบน PDF ไหม? | ความหมาย |
|---|---|---|
| `signer` | มี | ผู้ลงนามจริง ลายเซ็นปรากฏบนเอกสาร |
| `approver` | ไม่มี | อนุมัติ/ยืนยันเท่านั้น ไม่มีช่องเซ็น |
| `cc` | ไม่มี | รับสำเนาดูอย่างเดียว ไม่อยู่ในสายเซ็น ไม่มี `step` |

Badge สรุปแสดงเสมอ: *"ผู้ลงนาม: X · ผู้อนุมัติ: Y"* — ตัวเลขนี้ต้องตรงกับจำนวนช่องเซ็นจริงบน PDF เป๊ะๆ

## 1.3 ลำดับการเซ็น — Step (เรียงลำดับ & พร้อมกัน)
- ผู้ลงนาม/ผู้อนุมัติแต่ละคนมี `step` (เลขจำนวนเต็ม)
- **`step` เดียวกัน** → เซ็น **พร้อมกันได้**
- **`step` ต่างกัน** → ต้องเซ็น **ตามลำดับ** — คนที่ `step` ถัดไปถึงคิวได้ก็ต่อเมื่อทุกคนใน `step` ก่อนหน้า `signed` ครบแล้ว (คนที่ `rejected` ไม่บล็อกคิว แต่เอกสารทั้งฉบับหยุดทันที)
- `isMyTurn(doc, me)`:
  ```
  minPending = MIN(step ของทุกคนที่ยังไม่ signed & ยังไม่ rejected)
  isMyTurn = (step ของฉัน === minPending) และ (status ของ doc === 'progress')
  ```
- ก่อนบันทึก ระบบปรับ step ให้เรียงติดกัน 1..k เสมอ (`normalizeSteps`)
- `groupSignatories()` — จัดกลุ่มคนตาม step เพื่อแสดงผล "รอบที่ 1: A, B เซ็นพร้อมกัน → รอบที่ 2: C อนุมัติ"

## 1.4 การมอบหมาย (Delegation)
- ที่นั่ง (signer) 1 ตำแหน่ง มอบหมายให้คนอื่นเซ็นแทนได้ผ่านฟิลด์ `assignedTo`
- `actingId(signer) = signer.assignedTo || signer.id` — ทุกจุดที่ต้องหา "ใครต้อง action จริง" ต้องใช้ `actingId` เสมอ ห้ามใช้ `signer.id` ตรงๆ
- ตำแหน่งช่องเซ็น (`placements`) ยังผูกกับที่นั่งเดิม (`signerId` = id เดิม) ไม่เปลี่ยนตามผู้ได้รับมอบหมาย

## 1.5 ขั้นตอนการสร้างเอกสารใหม่ (3 ขั้นตอน)
**ขั้นตอนที่ 1 — Step1Input**: กรอกชื่อเรื่อง เลือกประเภทเอกสาร (docType/docSubtype หรือ "อื่นๆ" พิมพ์เอง) อัปโหลดไฟล์ PDF หลัก + ไฟล์แนบ
- ถ้าเลือก `docSubtype` ที่ตั้งค่าสายอนุมัติไว้แล้ว → ระบบสร้างรายชื่อผู้ลงนาม + CC อัตโนมัติตาม `subtypeRoute()` (ดูข้อ 1.8) อาจถูกล็อกไม่ให้แก้ถ้าตั้งเป็น `lockAll`
- เลขที่เอกสาร (`docNo`) แสดงตัวอย่างล่วงหน้า = `nextDocNo(...)` ต้องตรงกับเลขจริงตอน submit เป๊ะๆ

**ขั้นตอนที่ 2 — Step2Place**: ลาก-วางช่องเซ็น (`placements`) ลงบนแต่ละหน้า PDF ให้แต่ละ `signer` (approver/cc ไม่ต้องวางช่อง)

**ขั้นตอนที่ 3 — Step3Send**: ตรวจทานทั้งหมด → กดส่ง
- `status = 'progress'`, ทุก signer เริ่มที่ `status: 'pending'`, บันทึกเวลาสร้างจริง (real-time)

## 1.6 ขั้นตอนการลงนามเอกสาร — 3 วิธีเซ็น
เงื่อนไขเข้าหน้าเซ็นได้: `isMyTurn(doc, me) === true`

| วิธี | ปุ่ม | ต้อง OTP ไหม | เก็บข้อมูลยังไง |
|---|---|---|---|
| **A. ลายเซ็น Superwork** | ใช้ลายเซ็นที่บันทึกไว้ หรือวาดใหม่บน canvas | **ไม่ต้อง** | `sigType: 'img'` |
| **B. ลายเซ็นผ่าน Lanit** | ลายเซ็นดิจิทัลรับรอง (certified) | **ต้อง** (`otpFor='lanit'`) | `sigType: 'lanit'` |
| **C. ลายเซ็นต้นฉบับ** | พิมพ์ → เซ็นมือ → สแกน → อัปโหลดกลับ | **ต้อง** (`otpFor='original'`) | `sigType: 'original'`, เก็บแยกใน `signedVersions[]` |

**A. ลายเซ็น Superwork** (`useSuperwork()` → `startPlace()`):
1. เคยบันทึกลายเซ็นแล้ว → ใช้ทันที ไม่ต้องยืนยันเพิ่ม
2. ยังไม่เคย → เข้าหน้าวาดใหม่ → ถามว่าจะบันทึกไว้ใช้ครั้งหน้าไหม
3. เข้าหน้าวางตำแหน่ง → ยืนยัน — **ไม่มีขั้นตอน OTP**

**B. ลายเซ็นผ่าน Lanit** (`useLanit()` → `markSigned()`):
1. กด → จำลองเชื่อมต่อ → ส่ง OTP 6 หลัก (นับถอยหลัง 60 วิ ก่อนขอใหม่ได้)
2. กรอก OTP ครบ → สร้าง `sig { img, type:'lanit', sealImg, date, scale }` ผูกกับ placement
3. เข้าหน้าวางตำแหน่ง/ยืนยันเหมือนวิธี A

**A/B ปลายทางเดียวกัน (`markSigned`)**:
- `signer.status = 'signed'`, `signer.time = ตอนนี้`
- ทุกคน signed ครบ → `document.status = 'done'` ไม่ครบ → ยัง `progress`
- แจ้งเตือน: ผู้สร้างเอกสาร *"[ชื่อ] ได้เซ็น '[title]' แล้ว"*; จบครบทุกคน → แจ้งทุกคนที่เกี่ยวข้อง (ใช้ `actingId`); ยังไม่จบ → แจ้งทุกคนใน step ถัดไป ("อนุมัติ" ถ้าเป็น approver, "ลงนาม" ถ้าเป็น signer)

**C. ลายเซ็นต้นฉบับ / อัปโหลดสแกน** (`onOriginalSign()`):
1. ดาวน์โหลด/พิมพ์ไฟล์ต้นฉบับ (ไม่มีลายน้ำ) → เซ็นมือจริง
2. สแกน/ถ่ายรูป อัปโหลดให้ครบทุกไฟล์ → preview ก่อนยืนยัน
3. ยืนยัน OTP (`otpFor='original'`)
4. บันทึกฉบับที่เซ็นแล้วลง `signedVersions[]` (ไม่ทับต้นฉบับ)
5. `signer.status='signed'`, `signer.sigType='original'`
6. ลบช่องเซ็น (`placements`) ของที่นั่งนี้ออก (มีลายเซ็นจริงบนไฟล์สแกนแล้ว ไม่ต้อง overlay ซ้ำ)
7. Logic แจ้งเตือนเหมือน A/B ทุกประการ (`notifyAfterSign` ร่วมกัน)
8. ถ้ามีคนก่อนหน้าเซ็นวิธีนี้แล้ว → คนถัดไปทำงานบนไฟล์ฉบับล่าสุดที่เซ็นมือแล้ว (`lastSignedVer`) ไม่ใช่ไฟล์ต้นฉบับเดิม

## 1.7 การปฏิเสธ / ยกเลิกเอกสาร
- **ปฏิเสธ** (`onReject`, เฉพาะคนถึงคิว): ต้องกรอกเหตุผล → `signer.status='rejected'`, **เอกสารทั้งฉบับ** เปลี่ยนเป็น `status='rejected'` ทันที ล็อกทั้งฉบับ ห้ามเซ็นต่อ แม้ signer คนอื่นยังไม่ถึงคิว → แจ้งผู้สร้างพร้อมเหตุผล
- **ยกเลิก** (`onCancel`, เฉพาะผู้สร้าง): `status='cancelled'` + `cancelReason` → แจ้งผู้ลงนามทุกคน (ยกเว้นผู้สร้างเอง)

## 1.8 การตั้งค่าสายอนุมัติตามประเภทเอกสารย่อย (Tab 6 "Flow Signature Approval Setting")

- แต่ละ `docSubtype` มี `chain: [step, step, ...]` แต่ละ step เป็น 1 ใน 4 แบบ:

## 1.9 เอกสารลับ (Confidential)
- `docType === "เอกสารลับ"` → เห็นได้เฉพาะ: ผู้สร้าง, ผู้ลงนาม/ผู้อนุมัติทุกคนในสาย, ผู้ที่อยู่ใน CC เท่านั้น
- คนนอกรายชื่อ: ห้ามเห็นในลิสต์/ภาพรวม/รายงานใดๆ ทั้งสิ้น แม้ในหน้าสถิติรวมทั้งระบบ

## 1.10 การออกเลขที่เอกสาร (docNo)
- รูปแบบ: `{PREFIX}-{ปี}/{ลำดับ 3 หลัก}` เช่น `FIN-2026/001`
- PREFIX: `doc.docPrefix` ที่ตั้งไว้ตอนสร้าง → ไม่มีดูจาก `docSubtype` → ไม่มี fallback ไป 1 ใน 11 `docType` หลัก (prefix คงที่: GEN, CFD, HR, CTR, CON, FIN, BUD, PUR, AUD, OUT, MED)
- ลำดับนับแยกตามคู่ (PREFIX + ปี) รีเซ็ตเป็น 001 ทุกปีใหม่
- ต้องไม่ซ้ำและเรียงต่อเนื่องในขอบเขต PREFIX-ปี — backend ต้อง atomic กันตอนสร้างพร้อมกันหลายคำขอ

## 1.11 คอมเมนต์ / @mention ในเอกสาร
- @Mention แท็กได้ทุกคนในบริษัท จัดกลุ่มตามแผนก — คนที่ถูกแท็กแต่ยังไม่อยู่ใน `cc[]` จะถูกเพิ่มเข้าอัตโนมัติ (ได้สิทธิ์เห็นเอกสารถาวร)
- **ยกเว้นเอกสารลับ**: จำกัดเฉพาะคนในสายเดิม (ผู้สร้าง/ผู้เซ็น/CC เดิม) ห้าม mention คนนอกสาย และ **จะไม่เพิ่มเข้า CC อัตโนมัติ**
- คอมเมนต์แบบไม่ mention ใคร → แจ้งทุกคน: ผู้สร้าง + ผู้ลงนามทุกคนในเอกสารนี้
- แก้ไข/ลบคอมเมนต์ของตัวเองได้เสมอ

# ส่วนที่ 2 — โมดูล APPROVAL (ศูนย์อนุมัติ)

## 2.1 ขอบเขต
Approval Center คือหน้าที่ผู้อนุมัติเห็นและ action **คำขอของคนอื่น** (ไม่ใช่ของตัวเอง) ครอบคลุม **7 ประเภทคำขอ**:

| kind | ชื่อที่แสดง | data model | สายอนุมัติ |
|---|---|---|---|
| `esign` | ขอลายเซ็น | `docs` (Document) | `subtypeRoute()` ตาม docSubtype (ดูข้อ 1.8) |
| `ot` | โอที | `reqs.ot` (Request) | หัวหน้าแผนกผู้สร้างเท่านั้น — 1 ขั้น |
| `leave` | ลาพัก | `reqs.leave` (Request) | หัวหน้าแผนก → HR — 2 ขั้น |
| `offsite` | งานนอกสถานที่ | `reqs.offsite` (Request) | หัวหน้าแผนก → HR — 2 ขั้น |
| `booking` | จองห้อง/รถ/อุปกรณ์ | `reqs.booking` (Request) | หัวหน้าแผนก → HR — 2 ขั้น (ยังไม่มีฟอร์มสร้างในแอป — เห็นได้แต่ยังสร้างคำขอใหม่ไม่ได้) |
| `knowledge` | แบ่งปันความรู้ | `reqs.knowledge` (Request + draft) | หัวหน้าแผนก → HR — 2 ขั้น |
| `points` | คะแนน Workboard | `pointsReqs` (Points) | ผู้อำนวยการ (DIRECTOR) คนเดียว — 1 ขั้น (ไม่ใช้ approvalChain) |

Tab "ทั้งหมด" รวมทุกประเภทไว้ในลิสต์เดียว = ที่มาของคำว่า "8 หมวด" ใน README (7 ประเภทจริง + 1 tab รวม)

## 2.2 โครงสร้างข้อมูล `Request` (leave/offsite/ot/booking/knowledge)
```js
Request = {
  id, title, byId,
  note, detail?,             // เหตุผล (leave/ot) หรือสถานที่ (offsite, note=สถานที่, detail=เหตุผล)
  date, dateTo?,               // วันที่ต้องการใช้สิทธิ์ (ไม่ใช่วันที่สร้าง)
  from?, to?, hours?,
  activity?, tasks?: string[], dayType?, // เฉพาะ OT
  files?: File[],
  status: 'draft' | 'progress' | 'approved' | 'rejected' | 'cancelled', // 'draft' เฉพาะ knowledge
  approvedBy: string[],       // id ที่อนุมัติแล้วเรียงลำดับ
  reason?,
  createdAt,
  history: [{ type, by, time, reason? }],
  comments: [{ id, byId, time, text, parentId?, mentions? }],
}
```

## 2.3 การสร้างสายอนุมัติ (Approval Chain) — สำหรับ leave/offsite/ot/knowledge
ฟังก์ชัน `approvalChain(byId, kind)`:
- หา **หัวหน้าแผนก** (`rank==='head'`) ของแผนกเดียวกับผู้สร้าง → **ขั้นที่ 1**
- ถ้า `kind !== 'ot'` → เพิ่ม **HR** (id คงที่ `u1`) → **ขั้นที่ 2**
- `ot` → **ขั้นเดียว** (หัวหน้าแผนกอย่างเดียว) ไม่ผ่าน HR
- ตัดคนซ้ำออก (เช่นผู้สร้างเป็นหัวหน้า HR เอง จะไม่มี HR ซ้ำ)
- สายอนุมัติคำนวณแบบ **ไดนามิกตามแผนกผู้สร้าง** ไม่ hard-code ตาม user

**`points` ไม่ใช้ฟังก์ชันนี้** — ใช้ผู้อนุมัติคนเดียวคงที่ = `DIRECTOR` เสมอ ไม่ว่าใครเป็นผู้ขอ (1 ขั้นตอนเสมอ)

## 2.4 State machine (leave/offsite/ot/booking/knowledge/points)
```
[สร้างใหม่] → status = "progress", approvedBy = []
   │  (knowledge เท่านั้น: อาจเริ่มที่ status = "draft" ถ้าบันทึกร่างไว้ก่อน ยังไม่แจ้งใคร)
   │
   ├─ ผู้อนุมัติที่ถึงคิว กด "อนุมัติ"
   │     approvedBy.push(approverId)
   │     approvedBy.length < chain.length  → ยัง "progress" ส่งต่อคนถัดไป
   │     approvedBy.length == chain.length → "approved" (จบ)
   │
   ├─ ผู้อนุมัติที่ถึงคิว กด "ปฏิเสธ" (ต้องกรอกเหตุผล)
   │     status = "rejected" ทันที ไม่ต้องรอครบทุกขั้น
   │
   └─ เจ้าของคำขอ กด "ยกเลิก" (ทำได้เฉพาะตอน progress)
         status = "cancelled" + เหตุผล
```
- `currentApprover(r, kind)` = `chain[approvedBy.length]` ถ้า `status==='progress'` นอกนั้น `null`
- ปุ่มอนุมัติ/ปฏิเสธแสดงเมื่อ: `request.byId !== me` **และ** `status==='progress'` **และ** `currentApprover.id === me`
- ปุ่มยกเลิกแสดงเมื่อ: `request.byId === me` **และ** `status==='progress'`
- คนอื่น (ไม่ใช่ผู้สร้าง/ไม่ใช่ผู้อนุมัติคิวปัจจุบัน) → ดูอย่างเดียว เห็นข้อความ "รอ [ชื่อ] ([ตำแหน่ง]) อนุมัติ"
- `points` จบได้ในขั้นตอนเดียวเสมอ (ไม่มี "ขั้น 1/2, 2/2")

## 2.5 กฎการแจ้งเตือน
**ตอนสร้างคำขอใหม่** (`onCreateReq` / `onCreatePoints` / `onCreateKn`):
- แจ้ง **ผู้อนุมัติขั้นที่ 1** (ถ้าไม่ใช่คนเดียวกับผู้สร้าง): *"[ชื่อ] ส่งคำขอ ... รอการอนุมัติ"* (deep-link `{kind, id}` กดเปิดตรงได้)
- `knowledge` แบบ draft → **ไม่แจ้งใคร** จนกว่าจะกด "ส่งกวดสอบ" (`onSubmitKn`)

**ตอนมี action อนุมัติ/ปฏิเสธ** (`onReqAction` / `onPointsAction`):
- ยังไม่จบ (มีขั้นถัดไป): แจ้งผู้อนุมัติคนถัดไป *"ถึงคิวของท่านแล้ว — กรุณาอนุมัติคำขอ '[title]' (ขั้น X/Y)"* + แจ้งผู้สร้าง *"ผ่านการอนุมัติขั้น X/Y แล้ว — กำลังรอ [ชื่อขั้นถัดไป]"*
- จบแล้ว (อนุมัติครบ/ถูกปฏิเสธ): แจ้งผู้สร้าง *"ได้รับอนุมัติครบทุกขั้นแล้ว"* หรือ *"ถูกปฏิเสธ — [เหตุผล]"*

ทุก action (สร้าง/อนุมัติ/ปฏิเสธ/ยกเลิก) บันทึกลง `history[]` — ใช้แสดงในปุ่ม (i) ประวัติกิจกรรม (`ReqActivityHistory`)

## 2.6 คอมเมนต์ / @mention ในคำขอ
- คอมเมนต์ได้ตอน `status==='progress'`; จบแล้ว (approved/rejected/cancelled) → ล็อกคอมเมนต์ พร้อมข้อความบอกเหตุผลที่ล็อก
- มี **@mention** เจาะจงคน → แจ้งเฉพาะคนนั้น (+คนที่ถูกตอบกลับถ้าเป็น reply)
- **ไม่ mention ใคร** (คอมเมนต์ทั่วไป) → แจ้ง **ทุกคนที่เกี่ยวข้อง**: ผู้สร้างคำขอ + ผู้อนุมัติทุกคนในสาย (กันกรณีผู้สร้างคอมเมนต์เองแล้วไม่มีใครถูกแจ้งเลย)
- แก้ไข/ลบคอมเมนต์ของตัวเองได้
- `points`: ไม่มี concept CC/auto-add — แจ้งตรงระหว่างผู้ขอ ↔ DIRECTOR เท่านั้น

## 2.7 การตรวจสอบข้อมูลตอนสร้างฟอร์ม (ReqForm — leave/offsite/ot)
- `leave`: เลือกประเภทลา (ป่วย/กิจ/พักประจำปี/...) วันเวลาถูกต้อง (ชั่วโมงรวม > 0) เหตุผล ≥ 3 ตัวอักษร
- `offsite`: เหมือน leave + สถานที่ ≥ 2 ตัวอักษร
- `ot`: เลือกโครงการ → กิจกรรม → อย่างน้อย 1 งานย่อย เลือกประเภทวัน เหตุผล ≥ 3 ตัวอักษร ชั่วโมง OT เริ่มหลังเลิกงานเป็นค่าเริ่มต้น (18:00)
- ช่วงพักเที่ยง (12:00–13:00) ถูกหักออกอัตโนมัติจากชั่วโมงรวมถ้าช่วงที่ขอคาบเกี่ยว
- ไฟล์แนบ: ไม่บังคับ จำกัดชนิด `pdf/jpg/png/heic/webp/doc/docx` สูงสุด 10MB/ไฟล์
- ถ้าเลือกหลายวัน (`dateTo` ต่างจาก `date`) → ชั่วโมงรวม = จำนวนวัน × ชั่วโมง/วัน

## 2.8 การเรียงลำดับรายการ (ใช้ร่วมกันทุก tab)
`sortPendingFirst`:
1. คำขอที่กำลังรอดำเนินการ (`progress`) ขึ้นบนสุดเสมอ
2. ในกลุ่มเดียวกัน → เรียงตามวันที่สร้างล่าสุดขึ้นก่อน

## 2.9 หมายเหตุเฉพาะประเภท

**Knowledge (แบ่งปันความรู้):**
- มีสถานะพิเศษ `draft` — บันทึกร่างได้โดยไม่แจ้งใคร กด "ส่งกวดสอบ" (`onSubmitKn`) แล้วค่อยเข้าคิว `progress` และแจ้งผู้อนุมัติขั้นที่ 1
- หน้ารายละเอียดใช้ `KnowledgeDetailBody` (คนละ component จาก Request ทั่วไป)

**Points (คะแนน Workboard):**
- สร้างจากเมนู FAB (+) → `PointsRequest.jsx` (เลือก Project → Activity → Task, ระบุจำนวนคะแนน + เหตุผล)
- เก็บแยก store `pointsReqs` (คนละ state กับ `reqs`)
- อนุมัติขั้นเดียวโดย DIRECTOR คนเดียวเสมอ (ไม่ใช้ approvalChain)
- คำขอคะแนนของตัวเองก็ยังแสดงในหน้า Approval Center (ไม่ถูกกรอง `byId!==me` ออกเหมือนประเภทอื่น) เพราะยังไม่มีที่ให้ดูใน Request module

**Booking (จองห้อง/รถ/อุปกรณ์):**
- ใช้กลไก state machine + สายอนุมัติเดียวกับ leave/offsite (หัวหน้าแผนก → HR)
- ปัจจุบันแสดงผลได้อย่างเดียว — ยังไม่มีฟอร์มสร้างคำขอใหม่ในแอป (ใช้ seed data)

**e-Sign:**
- หน้ารายละเอียดใช้ `DocDetail.jsx` แยกทั้งหน้า (ไม่ใช้ component ร่วมกับ Request ทั่วไป) เพราะต้องแสดง PDF viewer + ช่องเซ็น
- Logic การเซ็น/ปฏิเสธ/มอบหมาย ดูส่วนที่ 1 ทั้งหมด

# ส่วนที่ 3 — โมดูล REQUEST (คำขอของฉัน)

## 3.1 ขอบเขต
รองรับ **3 ประเภทเท่านั้น**:

| kind | ชื่อที่แสดง | หมายเหตุ |
|---|---|---|
| `leave` | ลาพัก | ประเภทย่อย: ป่วย/กิจ/พักประจำปี/ดูแลครอบครัว/คลอด |
| `offsite` | งานนอกสถานที่ | มีฟิลด์สถานที่ (`note`) |
| `ot` | โอที | เลือกโครงการ → กิจกรรม → งานย่อย, มีประเภทวัน |

**ไม่รองรับในโมดูลนี้:** `booking` (ยังไม่มีฟอร์มสร้างในทั้งแอป), `knowledge` (มีโมดูลแยกของตัวเอง), `points` (สร้างผ่าน FAB แต่ดูผลได้ที่ Approval Center เท่านั้น ยังไม่มี tab ของตัวเองใน Request), `esign` (คนละโมดูลเต็มตัว ใช้ `docs` ไม่ใช่ `reqs`)

## 3.2 Flow การสร้างคำขอใหม่
```
กด FAB (+) → เลือกประเภท (leave/offsite/ot) → เปิดฟอร์ม ReqForm.jsx
  → กรอกข้อมูล (validation ตามข้อ 2.7)
  → กด "ส่งคำขอ" → onCreateReq(kind, data)
      - status = 'progress' ทันที (ไม่มี draft state ต่างจาก knowledge)
      - createdAt = เวลาจริงตอนสร้าง (ต่างจาก date = วันที่ต้องการใช้สิทธิ์)
      - แจ้งเตือนผู้อนุมัติขั้นที่ 1 ทันที
  → popup "ส่งคำขอสำเร็จ!"
```

## 3.3 Flow การดูรายการคำขอของตัวเอง
```
เปิด tile "Requests" จาก Dashboard
  → เลือก tab ย่อย: ลาพัก / งานนอกสถานที่ / OT
  → list = reqs[kind].filter(r => r.byId === me)   ← เห็นเฉพาะของตัวเองเท่านั้น
  → filter ย่อยตามสถานะ: ทั้งหมด/รอ/อนุมัติแล้ว/ปฏิเสธ/ยกเลิก
  → เรียงด้วย sortPendingFirst
```

## 3.4 หน้ารายละเอียดคำขอ (ของตัวเอง)
- Header มีปุ่ม **(i) ประวัติกิจกรรม** — เปิด modal `ReqActivityHistory` แสดง timeline: สร้าง → ส่งกวดสอบ → อนุมัติทีละขั้น → (จบ/ปฏิเสธ/ยกเลิก)
- เป็นเจ้าของคำขอเสมอ (`byId===me`) → **ไม่มีปุ่มอนุมัติ/ปฏิเสธ**
- ปุ่ม **"ยกเลิกคำขอ"** — แสดงเมื่อ `status==='progress'` ต้องกรอกเหตุผล → แจ้งผู้อนุมัติที่เคยได้รับ noti คิวนี้แล้วทุกคน (`onCancelReq`)

## 3.5 ความสัมพันธ์กับโมดูล Approval
คำขอที่สร้างจากโมดูลนี้ไปโผล่ซ้ำในโมดูล Approval Center ให้ผู้อนุมัติที่ถึงคิวเห็นและ action ได้ — **ทั้งสองโมดูลอ่าน/เขียนข้อมูลชุดเดียวกัน** (`reqs[kind]`) ต่างกันแค่ filter การมองเห็น:
- โมดูล Request: `r.byId === me`
- โมดูล Approval: `r.byId !== me`

## 3.6 คอมเมนต์ในคำขอ / การตรวจสอบข้อมูลฟอร์ม
ใช้กลไกเดียวกับส่วนที่ 2.6 และ 2.7 ทุกประการ ไม่มีความต่างเฉพาะโมดูลนี้

# ภาคผนวก — หมายเหตุสำคัญพบในโค้ด

- ระบบ OTP/LANIT ตอนนี้เป็นการจำลอง (ไม่ได้เรียก API จริง ไม่ส่ง SMS/email จริง) — ต้องเชื่อมต่อผู้ให้บริการ CA/LANIT จริงในเวอร์ชันใช้งานจริง
- "ตั้งค่าสายอนุมัติ" (Tab 6, ข้อ 1.8) ปัจจุบันอยู่ในหน้า "ตั้งค่า" (Settings) ของโมดูล e-Sign ไม่ได้แยกเป็น tab หลักในแถบ tab
